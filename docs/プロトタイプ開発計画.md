# dance, unleashed プロトタイプ開発計画

## 1. 目的
- 企画書の核である「144秒の四つ打ち」「1ミスで全損」「成功するほど群衆と音が膨らむ」を、ブラウザで短期間に検証できるプロトタイプとして実装する。
- プレイ後に以下を判断できる状態を目標とする。
  - 単調な入力でも、144秒を通す緊張感が維持されるか
  - ミスで144秒丸ごと失う厳しさが成立するか
  - キックの反復だけでグルーヴ感と解放感が成立するか

## 2. プロトタイプ方針
- プラットフォーム: PCブラウザ優先。スマートフォンブラウザは最低限プレイ可能にする。
- 技術: HTML + CSS + JavaScript（ES Modules）、Canvas 2D、Web Audio API。
- ゲーム性は最小限に絞る。
  - 判定対象はキック入力のみ
  - 人数増加はスコア計算ではなく演出タイムラインとして扱う
  - 他の音は自由に鳴らせても、成功失敗には影響させない
- 非採用:
  - サーバー通信
  - アカウント、ランキング、セーブ
  - 高品質なキャラクターアニメーション
  - 複数楽曲対応
- 実装優先順位:
  1. キック入力の気持ちよさ
  2. ミス時の即時リセット体験
  3. 群衆と音の段階的インフレ
  4. 補助パーカッションの気持ちよさ

## 3. プロトタイプで検証する仮説
1. BPM固定の単一入力でも、全損ルールが十分なテンションを生む。
2. キック継続に同期して群衆と音が膨らめば、複雑なスコア要素がなくても「自分が場を解放している」感覚が出る。
3. 補助パーカッションを自由入力にすると、ゲーム性を増やさず身体性だけを足せる。
4. ミス後に即座にランを再始動できれば、厳しさがあっても再挑戦の意欲を維持できる。

## 4. スコープ
### 含めるもの
- 1プレイ144秒のシングルプレイ
- 4拍カウントイン
- キック用の単一判定入力
- 補助パーカッション用の自由入力
- 成功/失敗判定、連続成功数、現在解放人数
- 群衆の増加演出
- 段階的なBGMトラック追加
- クリア表示と即時リスタート

### 含めないもの
- チュートリアルステージ
- 設定画面
- 難易度選択
- ネットワーク機能
- 本番品質のアート・サウンド制作

## 5. 前提仕様
- BPMは 125 に固定する。
- プレイ時間 144 秒で、アクティブ区間はちょうど 300 ビートになる。
- キック入力のみが判定対象になる。
- 補助パーカッション入力はいつ押してもよく、成功失敗に影響しない。
- 1回でもキックをミスすると、そのランは失敗として終了する。
- ミス時は残り時間を 144 秒に戻し、現在の解放人数・連続成功数・追加トラックを 0 に戻して最初からやり直す。
- 解放人数はスコア計算ではなく演出値として扱い、連続成功に応じた固定カーブで増える。
- ノーミスで 144 秒を走り切ったランをクリアとする。

## 6. 開発ステップ
### Step 1: 土台構築
- 単一HTMLで起動できるローカル実行環境を作る。
- ゲームループ、固定BPMタイマー、入力イベント管理を実装する。
- デバッグ表示でビート番号、入力誤差、判定結果、現在フェーズを確認できるようにする。

### Step 2: コアゲーム実装
- カウントイン、キック判定、ミス時のラン再開始を実装する。
- 連続成功に応じて進む固定の解放演出カーブを実装する。
- HUDに残り時間、現在人数、連続成功数を表示する。

### Step 3: 演出実装
- Canvas上で群衆の増加を視覚化する。
- フェーズごとの背景変化、カメラ揺れ、フラッシュを追加する。
- Web Audio APIでキックと段階的なループ追加を実装する。
- 補助パーカッション入力でスネア、ハットなどを任意に鳴らせるようにする。

### Step 4: プレイアビリティ調整
- 判定幅、解放演出カーブ、演出密度を調整する。
- PCキーボード、マウス、スマホタップで最低限テストする。
- 1ミスで144秒やり直しの重さが過剰なら、判定幅と再始動演出を優先調整する。
- 補助入力が主役を食うなら、音量と発音数を制限する。

## 9. リスクと対策
- Web Audio API の再生開始制限
  - 対策: 最初のユーザー入力で AudioContext を開始する。
- 端末ごとの入力遅延差
  - 対策: 判定幅を設定化し、視覚ガイドは音より先に出しすぎない。
- 群衆描画が重くなる
  - 対策: 個体数をそのまま描かず、代表点と密度表現で人数感を出す。
- 単調さが先に勝つ
  - 対策: 25秒ごとに見た目と音のレイヤーを必ず変化させる。
- 失敗時のやり直し負荷が高すぎる
  - 対策: 失敗から再スタートまでの遷移を短くし、再開操作を不要にする。
- 補助音がノイズになる
  - 対策: キックより小さい音量にし、同時発音数を制御する。

## 10. 完了条件
- 144秒を通してプレイできる。
- 成功と失敗の判定が視覚的・聴覚的に分かる。
- ミス時にラン全体がリセットされ、即座に新しい144秒へ入れる。
- キックを維持すると解放人数と音の盛り上がりが明確に増す。
- 補助パーカッションが判定に影響せず、任意で気持ちよく鳴らせる。
- ローカルブラウザで安定して動作する。
